为什么要进行配置分发

首先我们来理解一下一个应用的运行流程，对于一个可以正常运行的程序来说。最主要的数据就是两部分，一部分是可以被计算机理解并执行的代码，另一部分是配置和资源数据。对于配置数据来说，我们肯定希望配置和代码是分离的，配置数据不会影响到代码的运行，代码的运行也不会改变配置数据的值。这样当我们需要改动某个部分的时候，能够最大限度的减少对其他部分的影响从而规避潜在的风险。

哪些属于配置数据

通常来讲，配置数据是指那些在应用写死（即我们常说的硬编码，hardcoded）的数据，这些数据本应属于需要我们主动修改才会改变的值，不会随着程序运行而自动改变。

如何处理配置数据

处理配置数据最好的方式是在统一的入口进行配置数据的分发。根据场景的规模的不同，可用的分发方案也是多样的。可以通过DB、Http接口进行分发，也可以通过专用的配置文件进行分发，更下沉一点可以在逻辑处理文件的固定位置进行分发。对于不同的场景，其实这些方案都是合理的，在一些大中型项目中更多的情况下是综合使用了多种方案进行配置数据的抽离和分发。

通常来说，需要我们抽离并进行分发的数据有如下这么几类:

  • URL拼接片段

  • 展示给用户的提示或辅助字符串

  • 一些需要多次用到的常量

  • Code和中英文的映射关系

  • 数据库的访问信息、加密的Key

  • 不同开发环境下的启动参数等
